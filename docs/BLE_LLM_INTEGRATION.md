# HardwareOne — On-device LLM over BLE (firmware contract request)

What the Android app needs to drive the on-device LLM (`ENABLE_ONDEVICE_LLM`) over the BLE
command channel — model management + **streamed** chat. Same "wrap an existing capability as
`json` CLI commands" pattern already used for `status`/`devices`/`sensors`/`controls`/`battery`.

The app side is **already built** against this contract (a Chat page that polls for tokens); it
just needs these commands to exist on the device.

---

## 0. Why a new command shape

The existing `llmgenerate <prompt>` CLI is **blocking and buffered** — its token callback only
appends to a local string and it returns the whole (capped ~2 KB, 128-token, fixed-temp) result
at the end. Over BLE that's unusable: it ties up the channel for the entire (multi-second to
multi-minute) generation, would trip the app's silent-device watchdog, and gives no live tokens.

The **web** already does it right: async-start + poll-result (`POST /api/llm/generate` →
`{session}`, then poll `GET /api/llm/result?offset=…` → `{text,done,len}`), backed by the chat
layer (`chatBeginTurn`, `chatReadStream`/`chatGetStreamLen`). **Please expose that same flow as
CLI/BLE `json` commands.** The machinery exists — this is a thin wrapper.

---

## 1. Read / control commands

| Command | Returns | Notes |
|---|---|---|
| `llmstatus json` | `{"v":1,"state":"READY","model":"tiny.bin","tokPerSec":12.4,"error":""}` | `state ∈ UNLOADED\|LOADING\|READY\|GENERATING\|ERROR`. `model` = filename only. Extra fields (psramKB, contextUsed/Max) welcome but optional. |
| `llmmodels json` | `{"v":1,"models":["tiny.bin","small.bin"]}` | list of loadable model files |
| `llmload <name>` | short text | **exists** — fine as-is; app re-polls `llmstatus json` (LOADING→READY) |
| `llmunload` | short text | **exists** — fine as-is |
| `llmstop` | short text | **exists** — aborts the in-progress generation |

(`status json`'s `connectivity.llm` is also fine for availability/state, and the app already
reads it. `llmstatus json` is what the Chat page polls directly.)

## 2. Streamed generation (the core ask)

**Start (async, non-blocking):**
```
llmgenerate json <prompt>
  → {"v":1,"session":42}
  → on failure: {"v":1,"ok":false,"error":"model not ready"}
```
- Mirrors `POST /api/llm/generate`: kicks off generation in the background and **returns
  immediately** (do NOT block until generation finishes).
- The `json` token is the mode flag; **everything after it is the prompt** (strip the leading
  `json `). The prompt may contain spaces.
- Optional trailing args are fine to support later (`max_tokens`, `temp`), but not required for v1.

**Poll for tokens:**
```
llmresult json <offset>
  → {"v":1,"text":"…bytes since <offset>…","done":false,"len":137}
```
- Mirrors `GET /api/llm/result?offset=N`: `text` = generated bytes **since `offset`**, `len` =
  total bytes so far, `done` = generation finished.
- The app polls this ~every 350 ms, appending `text`, advancing `offset = len`, until
  `done:true`. (No `session` arg needed if there's one active generation; include/accept one if
  the chat layer wants it.)

That two-command loop is the whole streaming protocol.

---

## 3. App behaviour (so the shapes are motivated)

1. On the Chat page: `llmmodels json` once; poll `llmstatus json` ~every 2 s for state/model/tok-s.
2. Load a model from the picker → `llmload <name>` → watch `LOADING → READY`.
3. Send a prompt → `llmgenerate json <prompt>` → get `session`.
4. Poll `llmresult json <offset>` → stream tokens into the assistant bubble → until `done`.
5. **Stop** button → `llmstop`.
6. If `llmgenerate json` / `llmresult json` aren't recognized (non-JSON reply), the app shows
   "LLM streaming isn't supported by this firmware yet" — so partial firmware support degrades
   cleanly.

## 4. Auth / availability

- Requires an authenticated session (login), like the other commands.
- Availability is implicit: only built when `ENABLE_ONDEVICE_LLM`. The app already gates the LLM
  card on `status json`'s `connectivity.llm`; the Chat page reports "not available" if
  `llmstatus json` doesn't come back as JSON.

## 5a. Chat-management commands  **[implemented]**

Generation is **multi-turn** — `llmgenerate json` rides `chatBeginTurn`, so the device keeps the
conversation (`contextUsed`/`contextMax` grow across turns). The firmware now exposes the
chat-management ops over BLE (wrapping the same `chatClear`/`chatRetryLast`/turn-readers the web
uses):

- **`llmclear`** → `"Conversation cleared"` (plain text). Resets the device turn history.
  **Refused mid-generation** (`"Busy: stop generation first"`) → the app sends `llmstop` then
  `llmclear` on Clear.
- **`llmretry`** → `{"v":1,"session":N}` (or `{"v":1,"ok":false,"error":…}`). Drops the last
  assistant turn and regenerates — returns a session like `llmgenerate`, then poll
  `llmresult json <offset>` as usual. Wired to the chat header **Retry** button.
- **`llmturns json <index>`** → one turn per call: `{"v":1,"count":N,"index":i,"role":"user"|
  "assistant","text":"…","tokens":N,"tokPerSec":F,"streaming":bool}`; out-of-range →
  `{"v":1,"count":N,"index":i,"end":true}`. (Per-call indexing because a BLE reply is capped at
  4 KB and can't chunk an unbounded array.) **Not yet wired** — would let the app resync the
  conversation after an app restart / cross-interface (web↔app). Optional follow-up.

  ⚠ field name is **`tokPerSec`** (the firmware handoff had a `tokOerSec` typo in one example).

*FYI, no action:* the sampling-default setters exist as CLI verbs taking a value
(`llmtemperature <0-2>`, `llmmaxtokens <1-512>`, `llmtopp`, `llmmaxcontext`, …) — an LLM settings
panel can drive them directly later.

## 5. Implementation pointers (firmware side)

- `llmgenerate json` ≈ the `chatBeginTurn(prompt, …)` path the web handler already calls →
  return `{session}`.
- `llmresult json <offset>` ≈ `chatReadTurn(streamingIndex, offset, …)` / `chatGetStreamLen()` →
  `{text, done, len}`. `done` when the streaming turn is no longer generating
  (`chatIsGenerating()` false and offset == len).
- `llmstatus json` / `llmmodels json` ≈ serialize `llmGetStatus()` / the model-file list — same
  as `GET /api/llm/status` and `GET /api/llm/models`.
- Reuse `argWantsJson()` for the `json` flag, and the PSRAM-buffer / no-`String` discipline used
  by the other `json` commands.
